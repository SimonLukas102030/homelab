Software Stack

Proxmox VE as the hypervisor — VMs for the heavy lifting (Minecraft), LXC containers for the lightweight services (Plex, Nextcloud, Portainer). Everything behind a reverse proxy with a Cloudflare tunnel to simonsstudios.de.

Storage-Pool:
- local-zfs    NVMe 512 GB    Proxmox + VM/LXC roots + Plex-Cache
- bulk-zfs     HDD 2 TB       /mnt/media (Plex-Library) + /mnt/cloud (Nextcloud-Files) + Backups

Why VMs for MC, LXC for the rest? MC needs its own JVM instance with dedicated RAM, isolated crashes (one VM dies → the others stay up). Plex/Nextcloud gain little from VM isolation but benefit a lot from LXC efficiency (no kernel overhead, near-native speed, bind-mounts possible).

01Proxmox VE — Installation

  1. Get the ISO: proxmox.com/de/downloads → "Proxmox VE 8.x ISO Installer". Write to USB with Rufus or Ventoy.
  2. Enable in BIOS: SVM/AMD-V (CPU virtualization), IOMMU (for GPU passthrough later), Wake-on-LAN, S3 suspend.
  3. Boot from USB → graphical installer. Key points: filesystem ZFS (RAID0 single-disk) on NVMe — gives you snapshot support. Hostname: homelab. Static IP: matching your LAN (e.g. 192.168.178.50).
  4. First login: https://<ip>:8006 — user root, password as set during install.
  5. No-Subscription repo activation (otherwise the update dialog nags you). Datacenter → Subscription → Remove; in /etc/apt/sources.list.d/ remove the enterprise source, add the no-sub repo.
  6. Mount the HDD: Datacenter → Storage → create ZFS pool bulk-zfs from the 2 TB HDD, or mount directly as a directory under /mnt/bulk.
  7. Plan VMID ranges: 100–199 for VMs, 200–299 for LXC. Keep it consistent.
💡
ZFS on a single NVMe: overkill for data integrity (no redundancy), but gives you ZFS snapshots — you can take a pre-update snapshot and roll back in 2 seconds. Worth it for that alone.

02Minecraft Server (Paper, Java 21, Aikar's Flags)

One dedicated Ubuntu VM per MC server. 4 vCores is enough — MC is single-thread heavy, more cores don't help much. 16 GB RAM allocated, ~14 GB of that for the JVM heap. Java 21 is required for 1.20.5+.

VM Setup (Proxmox)

  1. Create VM: ID 100, OS Ubuntu 24.04 Server, 4 vCores (host CPU type), 16 GB RAM, 30 GB virtio-SCSI on local-zfs, 1× virtio-net bridge on vmbr0.
  2. Install Ubuntu, set static IP, enable SSH, install openjdk-21-jre-headless + screen or tmux.
  3. Get the Paper JAR: papermc.io/downloads/paper → latest 1.21.x. wget directly to /opt/mc/paper.jar.
  4. First run accepts the EULA: echo "eula=true" > eula.txt
  5. Start script with Aikar's Flags (see below). Set up as a systemd service or run via tmux.

Aikar's Flags (for 14 GB heap, MC server)

#!/bin/bash
java -Xms14G -Xmx14G \
  -XX:+UseG1GC -XX:+ParallelRefProcEnabled -XX:MaxGCPauseMillis=200 \
  -XX:+UnlockExperimentalVMOptions -XX:+DisableExplicitGC \
  -XX:+AlwaysPreTouch -XX:G1NewSizePercent=30 -XX:G1MaxNewSizePercent=40 \
  -XX:G1HeapRegionSize=8M -XX:G1ReservePercent=20 -XX:G1HeapWastePercent=5 \
  -XX:G1MixedGCCountTarget=4 -XX:InitiatingHeapOccupancyPercent=15 \
  -XX:G1MixedGCLiveThresholdPercent=90 -XX:G1RSetUpdatingPauseTimePercent=5 \
  -XX:SurvivorRatio=32 -XX:+PerfDisableSharedMem -XX:MaxTenuringThreshold=1 \
  -Dusing.aikars.flags=https://mcflags.emc.gs -Daikars.new.flags=true \
  -jar paper.jar nogui

Important: Xms == Xmx (same value). G1GC with the region sizes from Aikar's recommendations. With 16 GB VM RAM: heap at 14 GB → 2 GB left for Linux + JVM overhead.

Paper tuning via spigot.yml / paper-global.yml

  • view-distance: 8 (default 10 → reducing to 8 saves enormous RAM/CPU)
  • simulation-distance: 6 (mob tick range)
  • no-tick-view-distance: 12 (players see further, but chunks don't tick)
  • entity-activation-range: animals 24, monsters 32, misc 16 — works for moderate load

Monitoring

Install Spark as a plugin — gives you TPS tracking, CPU profiling, heap analysis directly in-game. TPS < 19 means something is wrong.

03Plex Media Server

Setup as LXC Container 2 vCores · 4 GB RAM

Running Plex officially in an LXC is efficient — near-native I/O, low RAM footprint. Ubuntu 24.04 LXC, then plexmediaserver from the official Plex repo.

Bind-mount for media: the Plex LXC sees /mnt/media from the host bulk pool. In /etc/pve/lxc/200.conf:

mp0: /mnt/bulk/media,mp=/mnt/media

Hardware transcoding: the Ryzen 7 5700X has no iGPU — transcoding runs in software. That's fine for 1–2 parallel 1080p streams (the CPU handles it easily), but 4K transcoding will be slow. Fix if needed: add a cheap Intel ARC A310 (~€120) as a transcoding card, pass it through via vfio to the LXC. For now: skip.

Plex Pass: if you pay for a Plex Pass (~€5/month or Lifetime ~€120), you get hardware transcoding support, mobile sync, intro skip, and Live TV. Without it everything still works, just no HW transcode.

04Nextcloud

Option A: Nextcloud AIO (Docker-in-LXC) recommended

Nextcloud AIO (All-in-One) is the official Docker setup with a master container that manages everything (Apache, database, Redis, Talk, Office). LXC with Docker inside, then AIO via docker run. Advantage: one-click updates via the AIO UI, all components stay consistent.

Storage: bind-mount /mnt/bulk/cloud as the Nextcloud data path. Files live on the HDD, not on the NVMe.

Option B: Native Nextcloud in LXC more control

Install Apache/PHP-FPM/MariaDB directly in the LXC. More tuning options, less magic. More maintenance overhead. Only choose this if you enjoy Linux sysadmin work.

🔐
HTTPS: Nextcloud complains loudly without TLS. Solution: Nginx Proxy Manager (LXC 203) with a wildcard cert via Cloudflare DNS challenge → cloud.simonsstudios.de gets a valid Let's Encrypt cert, internally it stays HTTP between NPM and Nextcloud.

05Portainer + Docker

Setup in LXC with Docker 2 vCores · 2 GB RAM

Ubuntu LXC, install Docker (official convenience script), launch Portainer CE as the first container. Portainer is a web UI for everything Docker-based — container management, stack deployment via Compose, logs, stats.

# Start Portainer CE
docker volume create portainer_data
docker run -d -p 9443:9443 --name portainer --restart=always \
  -v /var/run/docker.sock:/var/run/docker.sock \
  -v portainer_data:/data \
  portainer/portainer-ce:latest

What runs via Portainer: smaller services that don't justify their own LXC — e.g. uptime-kuma (uptime monitoring), vaultwarden (Bitwarden self-host), homepage (dashboard), changedetection.io (web page monitoring).

06Reverse Proxy (Nginx Proxy Manager)

NPM in LXC 1 vCore · 512 MB RAM

Nginx Proxy Manager is Nginx + a UI for SSL certificates. You click together subdomains (cloud.simonsstudios.de → 192.168.178.51:11000), NPM fetches a Let's Encrypt cert via Cloudflare DNS challenge (wildcard supported) and proxies the traffic.

Setup:

  1. LXC with Docker, deploy NPM container
  2. Create Cloudflare API token (DNS edit permissions only for simonsstudios.de)
  3. In NPM SSL → Add Wildcard Cert → Cloudflare DNS challenge → *.simonsstudios.de
  4. One proxy host per service: subdomain → backend IP:port → select cert

Alternatives: Caddy with the caddy-cloudflare plugin — config-as-code, less clicking, more sysadmin-oriented. Traefik if you want Docker-native (auto-discovery via labels).

07Backups

Proxmox Backup Server (PBS) as a secondary device ~€100 used (e.g. an old NUC)

Proxmox's own backup solution. PBS deduplicates at block level → backups for 2× MC + LXCs often only take 5–10 GB per full backup. Schedules via the Datacenter UI. If your main server dies, you have intact backups on the PBS.

Minimal option: Proxmox's own vzdump to the HDD in the same server — protects against software crashes, NOT hardware failure. Better than nothing.

Off-site: Important data (Nextcloud user files) additionally synced to Hetzner Storage Box (~€4/month for 1 TB) via rclone + cron.

⚠
3-2-1 rule: 3 copies of data, on 2 different media, 1 off-site. A local backup on the same HDD is ONLY protection against software crashes. A second server / NAS / cloud storage is required for real safety.

08Local AI (Medium/Pro Tier)

Ollama VM with GPU passthrough

In Medium and Pro tiers you have a GPU in the server. It gets passed directly into an Ubuntu VM via PCIe passthrough. There you run Ollama as the LLM engine (auto GPU detection) and Open WebUI as a ChatGPT-style frontend. Accessible via ai.simonsstudios.de through NPM.

What it can do: Llama 3 70B Q4 on 24 GB VRAM, Phi-4 14B (sweet spot, near ChatGPT quality), Stable Diffusion XL, Whisper for subtitles, local coding assistant via Continue.dev in VS Code.

Full setup details, GPU comparison, performance numbers, model selection: on the dedicated Local AI page.

09Monitoring

Three options, depending on how deep you want to go:

  • Netdata (LXC, single container) — installs everywhere instantly, nice UI, covers 95% of "what's going on with my server?". Recommended starting point.
  • Uptime Kuma (Docker via Portainer) — service uptime, HTTP checks, ping. "Are my MC servers reachable?".
  • Grafana + Prometheus + node_exporter — full stack, professional, significant setup effort. If you really want deep metrics (per-VM CPU, disk IOPS, network).

Current recommendation: Netdata + Uptime Kuma covers 99% of all needs. Grafana/Prometheus later if you feel like playing with visualization.
